# ADR-0003: .ofdx の未知キー扱い — acoustic 内ネスト追加のみ、保持機構は次期対応

## Status

Accepted (2026-07-16) — `opera_analysis` のネスト追加は実装済み。
未知キー保持機構は**非実装** (次期対応として記録)。

## Context

- `.ofdx` の現行 save (`OfdxIO::save`) は、メモリ上のモデルから
  **既知フィールドのみで JSON を再構成**する方式である。ロード側は
  「欠落キー = 既定値、未知キー = 無視」で前方互換だが、save 側には
  未知キーの**保持**機構がない。つまり別ツール (将来の新バージョン、
  外部スクリプト) が `.ofdx` に書いたキーは、本アプリで開いて保存
  した時点で消える。
- 対照的に `.ofd` (本家互換テキスト) は未知行を
  `Project::extraLines()` に保持し保存時に書き戻す (フェーズ0 調査
  §3.2) — 本家カーネル互換のための設計。
- 今回の `opera_analysis` 追加で選べた選択肢:
  - **案A**: 既存 `acoustic` オブジェクト内へのネスト追加のみ。
    未知キー保持は導入しない (現状維持)。
  - 案B: 「読み込んだ `acoustic` オブジェクトの未知キーをメモリに
    保持し、保存時に既知フィールドへマージして書き戻す」機構を導入。
  - 案C: `.ofdx` 全体を DOM (QJsonObject) のまま保持し、既知
    フィールドだけ差し替える。

## Decision

**今回は案A を採用する** (opera_analysis は `acoustic` 内の追加キー
のみ、未知キー保持機構は実装しない)。

**案B を次期対応として記録する**: 読み込み時に `acoustic` オブジェクト
の未知キー (および `opera_analysis` 内の未知キー) を
`QJsonObject` としてモデルに保持し、save 時に「保持した未知キー →
既知フィールドで上書き」の順でマージして書き出す。

案B/C を今回見送った理由 (リスクと工数):

1. **マージ規則の曖昧さ**: ネストしたオブジェクトの深いマージ
   (既知の子キーと未知の子キーが同居する場合) の規則を決め、全階層で
   一貫させる必要がある。中途半端な実装は「消える」より悪い
   「古い値が復活する」バグ (stale データの再保存) を生む。
2. **現時点で実害がない**: `.ofdx` を書く外部ツールはまだ存在せず、
   本アプリ自身は常に全既知キーを書く。旧バージョンとの往復では
   「新キーが消える」だけで、これは既定値で安全に復元される。
3. **テスト工数**: 保持機構は selftest への未知キー往復チェック追加が
   必須であり、フェーズ2 の主目的 (RirAnalysisTab) より優先度が低い。

## Consequences

- (+) 実装ゼロで後方・前方互換の基本 (欠落キー既定値・未知キー無視) は
  維持される。旧バージョンで 1.1 ファイルを開いても壊れない。
- (−) **既知の制限として明文化**: 新しいバージョンや外部ツールが書いた
  `.ofdx` キーは、現行バージョンで保存すると失われる。バージョン混在
  環境で `.ofdx` を共有するワークフローには現状不向き。
  (`docs/opera-acoustics-file-format.md` §3-2 にも明記。)
- (−) 将来案B を入れる際、それまでに増えたキー全部のマージ試験が
  必要になる (先送りコスト)。
- 運用ルール: それまでの間、`.ofdx` への機能追加は必ず「追加キーのみ・
  既存キー不変・enum 値並び替え禁止」を守ること (baseline 運用
  ルール 4 と同じ)。
